# Clair UI/UX 问题清单

## 1. Profile 创建后缺少可编辑能力

当前系统中，Profile 更像是一次性配置项。用户创建并配置好某个 Profile 后，后续没有明显入口对其进行修改。

### 1.1 用户无法调整已有 Profile 的核心配置

例如：

* Profile 名称
* 绑定的 Provider
* 使用的模型
* 路由地址
* 启动命令
* settings 文件路径
* 模型映射规则
* 相关环境变量或代理参数

### 1.2 用户容易产生"配置错了只能重建"的感知

如果用户第一次填写时出现错误，或者后续 Provider 地址、模型名称、命令路径发生变化，当前界面会让用户感觉只能删除重建。

### 1.3 Profile 的状态管理不够完整

当前 Profile 能被查看、测试、复制命令、重新生成 Wrapper，但缺少"编辑已有配置"的闭环。

---

## 2. Provider / Profile 头像视觉质量较弱

当前 MiniMax 和 GLM 的头像以简单字母块形式呈现，例如 `M`、`G`。

### 2.1 头像视觉质感偏弱

整体 UI 风格比较干净、现代，但头像部分显得比较临时，像是占位符。

### 2.2 Provider 品牌识别度不足

MiniMax、GLM 等 Provider 之间目前主要通过文字区分，头像没有形成足够清晰的品牌感。

### 2.3 用户缺少自定义头像的表达空间

当前头像是系统默认生成的，用户无法根据自己的 Provider 类型、模型用途、个人偏好进行自定义。

---

## 3. 中间区域信息密度偏低，空间利用不足

当前主内容区采用卡片式布局，一个 Provider 以一个小方片卡片展示。

### 3.1 大量横向空间被浪费

界面中部区域非常宽，但 Provider 卡片只占据了很小一部分。

### 3.2 卡片承载的信息有限

当前每个 Provider 卡片能展示的信息较少，无法展示运行状态、绑定 Profile、模型、测试状态、路由、启动器状态等完整信息。

### 3.3 一个 Provider 占一个小卡片，不利于快速浏览

当前视觉上每个 Provider 是独立方块，没有充分利用一整行空间。

### 3.4 页面主体区域和信息价值不匹配

中间区域是整个应用视觉权重最高的区域，但它实际展示的信息较少。

---

## 4. 右侧 Profile 详情区域信息过长，阅读负担较重

右侧面板目前承担了 Profile 详细信息、生成状态、操作按钮、运行证据等内容。

### 4.1 信息内容过多，导致面板需要滚动

Profile 详情包含较多字段，集中堆叠在右侧窄栏中。

### 4.2 右侧面板宽度有限，不适合承载长路径和长文本

路径和 URL 在右侧窄面板中容易换行、截断或形成大块灰色文本区域。

### 4.3 详情信息层级不够清晰

静态配置详情、操作按钮、运行状态和测试证据混在一起。

### 4.4 关键详情和辅助信息混在一起

Profile 名称、路由、命令是核心信息；运行时模型映射说明、测试记录、生成日志则更偏辅助信息。

### 4.5 滚动区域影响操作效率

操作按钮和详情、日志混在一起，降低操作效率。

---

## 5. Profile 列表、Provider 卡片、右侧详情之间的关系不够直观

### 5.1 左侧选中的是 Profile，但中间展示的是 Provider

用户点击左侧的 Profile 后，中间仍然展示 Provider 卡片，造成认知割裂。

### 5.2 中间卡片和右侧详情的对应关系不够强

右侧展示的是 Profile 详情，但中间卡片展示的是 Provider 概览。

### 5.3 "Provider"和"Profile"的产品概念需要更明确

Provider 是上游模型服务配置，Profile 是可启动的 Claude Code 命令配置，两者关系没有充分表达。

---

## 6. 当前界面更像"配置结果展示"，而不是"长期管理工具"

* Profile 只能看，不能明显修改
* Provider 信息展示较轻
* 中间主区域承载的信息较少
* 右侧详情承载的信息过重
* 运行证据、配置详情、操作按钮混合在一起
* 用户难以快速完成"查看状态 → 定位问题 → 修改配置 → 测试验证"的完整管理流程
